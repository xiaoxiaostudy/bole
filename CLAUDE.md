# 项目协作要素

## 🚨 执行流程（最重要）

**非平凡的任务必须先对齐方案再动手。**

每次收到需求时：

1. **先沟通方案**：说清楚「我打算怎么做」「会改哪些文件」「可能的副作用」「有没有别的选择」
2. **等用户确认**：用户说 OK、说「开干」或明确同意之后再动手
3. **才开始写代码**：按照对齐后的方案执行

即便处于 Auto Mode / 自动模式，**这条规则也优先于自动执行**。用户偏好 > 自动模式默认行为。

### 适用范围

- ✅ **必须先确认**：改核心业务逻辑、新增/删除/重命名文件、修改数据模型、修改 LLM prompt、修改 API 契约、改排名/过滤算法、改 UI 主要结构
- ⚠️ **可以直接做，但要说一声**：修 typo、补 i18n、加 tests、调样式颜色/间距、重命名变量
- 🟢 **无需确认**：纯读操作（search、grep、read file）、类型检查、跑测试

### 模板

动手前用这个格式确认：

```
## 我打算这样做
1. <步骤 1>
2. <步骤 2>
...

## 会改的文件
- src/xxx.ts: <改什么>
- src/yyy.tsx: <改什么>

## 需要你先确认的点
- <开放决策 1>
- <开放决策 2>

OK 我就开始。
```

### 反面例子（别这样）

- ❌ 用户说「我觉得 X 有问题」→ 我立即去改代码
- ❌ 用户说「可以改成 Y 吗」→ 我当成指令立即执行
- ❌ 用户问「A 和 B 哪个好」→ 我直接选一个实现了

### 正面例子（这样做）

- ✅ 用户说「我觉得 X 有问题」→ 我先复述问题 + 提 2-3 个方案 + 问用户选哪个
- ✅ 用户说「可以改成 Y 吗」→ 我先说「可以，这里是影响范围 + 执行步骤」再问 OK 吗
- ✅ 用户问「A 和 B 哪个好」→ 我对比分析给出倾向，等用户拍板再动手

## 项目上下文

**名称**：LLM 决策报告系统（Beta）
**栈**：Next.js 16 (App Router) + React 19 + TypeScript + Tailwind v4 + Prisma (SQLite)
**主入口**：
- 首页 `/` — 需求描述 + 自适应追问
- 筛选页（内嵌在首页流程中）
- `/report/new` — 决策报告
- `/models` — 模型库浏览
- `/admin/*` — 管理后台

**核心数据流**：
用户描述 → elicitation LLM 抽取 RequirementVector → filter-by-vector 硬过滤 → rank 纯能力打分 → assembleReport → cost_weight>0 时软重排 → generate-report LLM 生成描述 → 渲染

**关键约定**：
- 预算是硬过滤（10% 容差），不是软打分
- `cost_weight` 只在 elicitation 阶段由 LLM 决定，不在 generate-report 里
- shortlist 排序必须在 LLM 生成描述**之前**锁定
- 数据源只有 AA 和 HF（arena 已弃用，不展示）
